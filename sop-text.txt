1. PURPOSE & SCOPE
This Standard Operating Procedure (SOP) defines the processes, workflows, and technical specifications for
interacting with the ACOB Odyssey API ΓÇö a production-grade metering and token management platform. This
document serves as the authoritative reference for development teams, operations staff, and system integrators.
1.1 Purpose
ΓÇó Define authentication protocols and JWT token management procedures
ΓÇó Document all available API endpoints, their parameters, and expected responses
ΓÇó Establish workflows for dashboard data, token operations, remote meter control, and reporting
ΓÇó Provide integration patterns for frontend dashboards and backend services
ΓÇó Specify error handling, retry logic, and monitoring procedures
1.2 Scope
ΓÇó API Base URL: http://8.208.16.168:9310
ΓÇó Platform: ACOB Odyssey Metering System
ΓÇó Applicable Modules: Dashboard, Token Management, Remote Operations, Data Reports, and Management
ΓÇó Audience: Software architects, backend engineers, frontend developers, DevOps, and system administrators
ΓÇó Sites Covered: KYAKALE, MUSHA, UMAISHA, TUNGA, OGUFA

JSON format. The API supports date-range filtering using ISO 8601 timestamps and site-based data segmentation
via SITE_ID parameters.

2. AUTHENTICATION & TOKEN MANAGEMENT
2.1 JWT Bearer Token
The Odyssey API uses JSON Web Tokens (JWT) signed with HS256. Every API request must include a valid Bearer
token in the Authorization header. Tokens contain role-based claims that control which endpoints can be accessed.
2.1.1 Request Header Format
Authorization: Bearer <JWT_TOKEN>
Content-Type: application/json
2.1.2 Token Claims

2.2 Security Best Practices
1. NEVER hardcode JWT tokens in source code. Store in environment variables or a secrets manager (AWS
Secrets Manager, HashiCorp Vault).
2. Validate token expiration (exp claim) before every API call. Refresh proactively 60 seconds before expiry.
3. Use HTTPS/TLS in production. The current HTTP configuration is NOT suitable for production.
4. Implement token rotation ΓÇö short-lived access tokens (15-60 min) with refresh tokens.
5. Log all token usage with user ID, timestamp, endpoint, and IP for audit trail compliance.
6. Implement rate limiting on your integration layer to prevent API abuse.

4.2 SOP: Credit Token Generation
Credit tokens are 20-digit STS codes dispensed to prepayment meters when a customer purchases electricity. Follow
this procedure for every credit token request.
1. Validate customer account status via Management.Customer endpoint before generating token.
2. Verify meter serial number is registered and active on the SITE_ID.
3. Calculate credit amount based on current tariff (retrieve from Management.Tariff endpoint).
4. POST to /api/token/creditToken with required payload fields.
5. Store generated token in TokenRecord immediately for audit and reversal capability.
6. Return 20-digit token to vending system / customer interface.
7. Log transaction with timestamp, amount, meter serial, and operator ID.
Credit Token Request Payload
{ "MeterSN": "METER_SERIAL_NUMBER",
"Amount": 5000,
"TariffRate": 25.0,
"SITE_ID": "{SITE_ID}",
"OperatorId": "USER_001" }

4.5 SOP: Token Cancellation / Reversal
1. Identify the token via the TokenRecord.CreditTokenRecord endpoint using date range and meter serial.
2. Verify the token has NOT been entered into the meter (check meter status via Remote Reading task).
3. POST cancellation request to the cancel endpoint with the original token reference.
4. Update TokenRecord with cancellation status, operator ID, and timestamp.
5. Issue reversal credit or customer notification as per business process.

5. REMOTE OPERATIONS ΓÇö SOP & API PROCEDURES
Remote operations enable server-initiated communication with meters through the GPRS/network gateway. All
operations are implemented as asynchronous tasks with status tracking.
IMPORTANT: Asynchronous Execution
Remote tasks do not execute synchronously. After creating a task, poll the task record endpoint until status
changes from PENDING to SUCCESS or FAILED. Typical response times range from 30 seconds to several
minutes depending on GPRS connectivity.
5.1 Meter Reading Operations
5.1.1 Create Meter Reading Task
POST /api/RemoteMeterTask/createReadingTask
{ "MeterSN": "METER_SERIAL", "ReadingType": "INSTANT", "SITE_ID": "{SITE_ID}" }
5.1.2 Retrieve Reading Task Result
GET /api/RemoteMeterTaskRecord/getReadingTask?TaskId=<TASK_ID>&SITE;_ID={SITE_ID}
5.1.3 Reading Task Polling SOP
1. POST create reading task. Store the returned TaskId.
2. Wait 30 seconds before first poll.
3. GET task record. Check status: PENDING / SUCCESS / FAILED / TIMEOUT.
4. If PENDING, wait 30 seconds and retry. Maximum 10 retries (5 minutes total).
5. If SUCCESS, extract meter reading data from response payload.
6. If FAILED or TIMEOUT, log error and trigger operator alert for manual investigation.
5.2 Meter Control Operations
Meter control tasks allow remote connect/disconnect and relay control of meters. These operations require strict
authorization and audit procedures.
Create Control Task
POST /api/RemoteMeterTask/createControlTask
{ "MeterSN": "METER_SERIAL", "ControlType": "DISCONNECT",
"Reason": "Non-payment", "AuthorizedBy": "OPERATOR_ID", "SITE_ID": "{SITE_ID}" }
MANDATORY: Dual Authorization for Disconnect

8.1 Exponential Backoff Implementation
async function apiCallWithRetry(endpoint, options, maxRetries = 3) {
for (let attempt = 1; attempt <= maxRetries; attempt++) {
try {
const response = await fetch(endpoint, options);
if (response.ok) return response.json();
if (response.status === 401) { await refreshToken(); continue; }
if (response.status < 500) throw new Error(`Client error: ${response.status}`);
} catch (err) {
const delay = Math.pow(2, attempt) * 1000; // 2s, 4s, 8s
if (attempt === maxRetries) throw err;
await new Promise(r => setTimeout(r, delay));
}
}
}

11. SECURITY & COMPLIANCE
11.1 Production Security Checklist
1. Migrate API base URL from HTTP to HTTPS. Implement TLS 1.2+ with valid certificate.
2. Store all JWT tokens in a secrets manager ΓÇö never in .env files committed to git.
3. Implement API Gateway with JWT validation as the first layer of defense.
4. Enable API request logging with correlation IDs for complete audit trail.
5. Implement rate limiting: 100 requests/minute per token; burst limit 20/second.
6. Restrict JWT token scopes to minimum required per integration (least privilege).
7. Rotate JWT tokens every 30 days. Immediately revoke tokens for departed staff.
8. Encrypt all sensitive data at rest using AES-256 (token values, customer data).
9. Conduct quarterly security review of API access logs and token usage patterns.
10. Implement IP allowlisting for production API access where possible.
11.2 Audit & Compliance Requirements
ΓÇó All token generation events must be logged with: timestamp, operator ID, meter SN, amount, and token hash.
ΓÇó Remote disconnect events require dual authorization records before execution.
ΓÇó Tariff changes must be logged with effective date, previous rate, new rate, and authorizing supervisor.
ΓÇó Customer data must be handled in accordance with applicable data protection regulations.
ΓÇó Retain API audit logs for minimum 7 years for regulatory compliance.
